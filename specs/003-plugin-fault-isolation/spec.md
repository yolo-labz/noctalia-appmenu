# Specification: Plugin fault-isolation envelope

**ID:** 003-plugin-fault-isolation
**Created:** 2026-05-10
**Author:** @phsb5321
**Constitution version:** 1.0.0

## Why

Pedro's bar today is a single noctalia-shell PanelWindow hosting many widgets (Launcher, Workspaces, Clock, Tray, …) plus our `noctalia-appmenu` BarWidget plugin. After 19 alphas of menubar work Pedro reported the same observation twice in two days: "When the appmenu does *anything* unusual, the rest of the bar stops working." Two concrete instances both already in flight as fixes:

- Bar-wide click freeze whenever the appmenu's submenu was visible (PR #52 — input-grab propagation via `xdg_popup.grab(wl_seat)`).
- Full-screen flicker on every focus change (PR #51 — surface-damage propagation through noctalia v4's single shared PanelWindow surface).

Both were diagnosed as *propagation paths* the host cannot turn off and the plugin must avoid. This spec turns those one-off fixes into a written fault-isolation contract: any future bug we ship must NOT be allowed to escape the appmenu's slot, and we need named invariants and verifiable checks so contributors (Pedro, future agents, anyone forking) cannot regress isolation without a CI failure.

The cost of skipping this is high. Pedro's alpha-rate frustration ("19 alphas") was driven entirely by isolation regressions — every bug felt 10× worse because it broke neighbouring widgets, not just the appmenu. Comparable projects (Plasma plasmoids, GNOME 45+ shell extensions) treat plugin fault-isolation as a first-class concern. We adopt the same discipline.

## User scenarios

### Scenario 1: appmenu has no menu data, bar stays clickable

**Given** the bridge has not yet pushed any menu data (e.g. fresh boot, focused app exposes no AT-SPI menu)
**When** the user clicks the Launcher / Workspaces / Clock / Tray
**Then** every other bar widget responds to clicks in <100 ms; the appmenu's slot is invisible (opacity 0) but the layout is stable; no surface damage is generated by the appmenu

### Scenario 2: appmenu submenu is open, bar stays clickable

**Given** the appmenu top-level "File" button is clicked and a dropdown is visible
**When** the user clicks Launcher (or any other bar widget that is not the appmenu)
**Then** the click is received by Launcher and the dropdown closes; no `xdg_popup.grab(wl_seat)` is in flight while the dropdown is up; verifiable via `WAYLAND_DEBUG=1 qs` ⇒ zero `popup.grab` lines for the appmenu surface, exactly one `set_keyboard_interactivity(none)` for the popup window

### Scenario 3: appmenu QML throws inside `applySnapshot`

**Given** the bridge pushes a malformed snapshot that triggers a JavaScript exception in `applySnapshot` (or inside a Repeater delegate, FileView callback, IpcHandler handler)
**When** the exception fires
**Then** the appmenu enters a defined "failed" state (visible: false, opacity: 0, slot reserved); the host bar's Repeater / Loader does NOT rebuild; sibling widgets are unaffected; an error is logged via `console.error("[appmenu] envelope caught:", …)` to `journalctl --user -u noctalia-shell`; subsequent valid snapshots can recover the widget without restarting the shell

### Scenario 4: bridge daemon crashes

**Given** `noctalia-appmenu-bridge` (the Rust daemon) crashes (e.g. niri-IPC drop, AT-SPI bus disappears, OOM)
**When** the bridge stops writing to `~/.cache/noctalia-appmenu/active.json` and stops calling the IpcHandler push channel
**Then** the appmenu retains the last successfully-rendered menu for at most one focus-change interval (≤2 s), then collapses to its zero-paint stable slot; sibling widgets continue to function; systemd `Restart=on-failure` brings the bridge back; on bridge return the widget recovers automatically

### Scenario 5: focus storm

**Given** the user alt-tabs rapidly (≥10 focus changes in 1 s) — a torture-test for state churn
**When** each focus change fires a `WindowFocusChanged` event through the bridge
**Then** the appmenu coalesces to at most one render per ~50 ms (debounce); identity-stable `topLevel` guard skips delegate rebuild when the focused app's menu structure is unchanged; bar reflow events from the appmenu are zero (stable-slot invariant); other bar widgets continue to receive their own input events at unmodified rates

## Functional requirements

### Surface-damage isolation

- **FR-001** The appmenu BarWidget MUST claim a stable `implicitWidth` (`maxStripWidth + margins`) at all times when `reserveSlot` is true (default). The implicit width MUST NOT vary based on whether a menu is currently rendered. Verifiable by reading the property in QML: with reserveSlot=true, `implicitWidth` is constant across the lifetime of the widget.
- **FR-002** Any animation on the appmenu (opacity fade, width morph) MUST occur INSIDE the reserved slot — no animation may extend the widget's geometry outside the slot bounds. Verifiable by `Behavior on implicitWidth { … }` only firing when `reserveSlot=false`.
- **FR-003** The appmenu MUST NOT use `visible: false` on its root `Item` while `opacity > 0` — the QML element MUST remain in the parent layout tree until any fade animation completes. (Prevents host-side relayout while the appmenu is mid-fade.)
- **FR-004** Top-level menu strip layout MUST clip overflow: child Row width is bounded by the slot's `width - margins`; longer menu strips elide rather than overflow.

### Input-grab isolation

- **FR-005** The appmenu's dropdown MUST be hosted on a sibling top-level `PanelWindow` with `WlrLayershell.layer: WlrLayer.Top`, `WlrLayershell.keyboardFocus: WlrKeyboardFocus.None`, and `WlrLayershell.exclusionMode: ExclusionMode.Ignore`. (Already implemented by PR #52.) Verifiable via `WAYLAND_DEBUG=1 qs ⇒ exactly one set_keyboard_interactivity(none) per screen for "noctalia-appmenu-popup-*" namespace, zero popup.grab calls.
- **FR-006** The dropdown's outside-click MUST be handled by a full-screen MouseArea inside the popup `PanelWindow`, NOT by `xdg_popup.grab` (which would steal the seat from the bar). Verifiable in `AppmenuPopupWindow.qml`: a `MouseArea { anchors.fill: parent; onClicked: root.close() }` exists at the popup root.
- **FR-007** Nested submenus (TODO post-v0.3.0) SHALL use `PopupWindow` parented to the new sibling `AppmenuPopupWindow`'s menu rectangle, NOT to the bar's `PanelWindow`. (Out-of-grab-scope for the bar; in-grab-scope for the parent popup, which is correct UX.)

### JS-error isolation

- **FR-008** Every public-callable entry point on the appmenu BarWidget root (`applySnapshot`, `IpcHandler.update`, `FileView.onLoaded`) MUST be wrapped in a `try { … } catch (e) { console.error("[appmenu] envelope caught:", e); root._failedState = true; }` envelope.
- **FR-009** When `_failedState === true`, the widget MUST render exactly the zero-paint stable slot (opacity 0, slot reserved) until either (a) `applySnapshot(null)` is called to clear, or (b) a fresh well-formed snapshot resets `_failedState = false`.
- **FR-010** Exceptions raised inside Repeater delegates MUST be contained by per-delegate try/catch around any user-facing action handler (`onClicked`, `Component.onCompleted`). A single broken delegate must not block siblings from rendering.

### Resource-budget isolation

- **FR-011** The appmenu MUST NOT use any QML `Timer` with `repeat: true` that fires more often than every 200 ms. Polled state (e.g. `Timer { interval: 50 }`) is forbidden in this widget.
- **FR-012** The appmenu MUST debounce upstream signal storms: at most one `applySnapshot` call per 50 ms is allowed to actually mutate widget state; intermediate calls coalesce to the latest payload.
- **FR-013** The appmenu MUST NOT allocate unbounded JavaScript objects in its handlers — every menu-tree object reference must be released when the source `topLevel` is replaced. Verifiable by `qs --memory-leak-trace` shows no growth across 10⁵ focus events.

### Dependency isolation

- **FR-014** The appmenu MUST function (in zero-paint slot mode) when `~/.cache/noctalia-appmenu/active.json` does not exist; widget MUST NOT throw or block load.
- **FR-015** The appmenu MUST function (in zero-paint slot mode) when the IpcHandler target name `appmenu` is not registered (bridge not started yet); the widget loads successfully and waits.
- **FR-016** The appmenu MUST recover automatically when the bridge restarts: within 2 s of the next bridge push (FileView watch + IpcHandler), the widget reflects the new snapshot, no shell restart required.
- **FR-017** Bridge-process death (SIGSEGV, OOM, etc.) MUST NOT propagate to the QML side; the widget continues running in zero-paint slot mode pending bridge respawn.

## Non-functional requirements

- **NFR-001 Performance** Each focus change emits at most one `applySnapshot` call (after coalescing); snapshot-to-render latency is ≤200 ms p95 on Pedro's hardware (AMD 7950X3D + Radeon 7800 XT).
- **NFR-002 Reliability** Three Quickshell sessions in a row, each running the bar for ≥4 hours with the integration harness's 5 reference apps (ghostty / Firefox / GIMP / VS Code / libreoffice), must show zero shell crashes and zero "rest of bar unresponsive" reports from Pedro.
- **NFR-003 Observability** Every isolation-related code path logs to `journalctl --user -u noctalia-shell` with the `[appmenu]` prefix, including: envelope catches, dependency-missing fallbacks, debounce-coalesced events, popup window state transitions.
- **NFR-004 Verifiability** Every functional requirement above has a corresponding test or runtime check: QML state assertion, WAYLAND_DEBUG grep, qmllint rule, Cargo test, or integration-harness script.

## Out of scope

- Architecture migration away from Quickshell or noctalia (v3 swarm verdict reaffirmed: stay).
- Aesthetic / animation changes to the appmenu beyond what isolation requires.
- Plugin manifest schema changes.
- Multi-screen-specific edge cases (already handled by per-screen BarWidget instances; no new requirement here).
- Submenu (popup-of-popup) behaviour — separate v0.3.1 work.
- Replacing the existing AT-SPI walker with a different menu source.

## Constraints / dependencies

- noctalia-shell v4 has a single full-screen PanelWindow architecture (issue #2216, "addressed in v5"). Isolation work assumes v4 host; some invariants (FR-001..FR-004) become trivial under v5 but the spec still describes correct widget behaviour.
- Quickshell PopupWindow's `xdg_popup.grab` semantic is a Wayland protocol property, not a Quickshell bug. FR-005..FR-007 are the protocol-level workaround.
- niri-IPC and AT-SPI buses are external dependencies the bridge subscribes to; their reliability is outside this spec but their failure modes drive FR-014..FR-017.
- Existing PRs #51, #52, #53, #54 are in flight and partially deliver this spec; they remain the canonical references.

## Success criteria

- **SC-001** Pedro can click any non-appmenu bar widget at any moment, regardless of appmenu state. Verified by manual test: open the appmenu submenu, immediately click Launcher, observe Launcher opens. Pedro signs off.
- **SC-002** Bar surface receives zero damage events from the appmenu during steady-state focus tracking (no menu changes, identity-stable topLevel). Verified by instrumenting noctalia-shell with `wl_surface.damage_buffer` logging for one minute.
- **SC-003** A deliberately-broken snapshot (malformed JSON, missing fields, throw-on-access) drops the appmenu into `_failedState`; the rest of the bar continues to function. Test fixture committed to `bridge/tests/fixtures/malformed/`.
- **SC-004** Bridge SIGKILL during steady-state operation does NOT freeze, repaint, or crash the QML host; the appmenu collapses to zero-paint slot within 2 s. Test runs as a CI integration step.
- **SC-005** No isolation regression survives `cargo test`, `qmllint`, OR the new integration harness. Each FR has a corresponding test artefact in CI.
- **SC-006** v0.3.0 final tag does not ship until SC-001..SC-005 all pass + manual sign-off from Pedro on Scenario 1..Scenario 5.

## Key entities

- **BarWidget root Item** — The `Item` declared by `plugin/BarWidget.qml`, host of the strip + popup + state. Owns `_failedState`, `topLevel`, `appId`, etc.
- **AppmenuPopupWindow (sibling top-level)** — `plugin/AppmenuPopupWindow.qml` declared as a child of BarWidget root; instantiated as a layer-shell sibling of the bar's PanelWindow per screen. Hosts the dropdown menu rectangle.
- **Active snapshot file** — `~/.cache/noctalia-appmenu/active.json`, written by the bridge daemon. Schema versioned (`"v": 1` per PR #55).
- **IpcHandler target "appmenu"** — push channel for steady-state updates from the bridge to the QML widget.
- **Failed state flag** — `root._failedState` boolean, true when the last `applySnapshot`/IpcHandler.update threw; reset on next valid snapshot.
- **Bridge daemon** — `noctalia-appmenu-bridge` Rust process, supervised by systemd user unit; writes the snapshot file and pushes via IpcHandler.

## Assumptions

- noctalia-shell v4 host is the deployment target; v5 migration is a separate concern handled by upstream.
- The host bar continues to use a Repeater over `BarSection.widgets`; isolation invariants assume that Repeater rebuilds are observable (so we can assert "rebuild does not happen" under steady state).
- Pedro's hardware (Radeon 7800 XT, AMD 7950X3D) is the worst-case stress profile for surface-damage propagation; passing on his desktop is the gate.
- The 5-app integration harness (ghostty, Firefox, GIMP, VS Code, libreoffice) is the canonical correctness test set per the v3 swarm synthesis.
